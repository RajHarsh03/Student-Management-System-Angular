# Requirements Document

## Introduction

The Student Management Module is a feature of the Angular Student Management System capstone project. It provides administrators with the ability to create, read, update, delete, and search student records through a web interface backed by a simulated REST API. The module lives under `src/app/students/`, consumes a shared service in `src/app/services/`, and uses a data model defined in `src/app/models/`. It integrates with Angular Routing, Reactive Forms, HttpClient, and RxJS, and co-exists with the Auth, Dashboard, and Shared/UI modules built by other team members.

---

## Glossary

- **Student_Module**: The Angular feature module located at `src/app/students/` responsible for all student-related views and logic.
- **Student_Service**: The injectable Angular service located at `src/app/services/student.service.ts` that encapsulates all HTTP calls for student data.
- **Mock_API**: The simulated REST back-end (e.g., json-server or Angular in-memory-web-api) that responds to GET, POST, PUT, and DELETE requests at the `/students` endpoint.
- **Student**: A data entity with the fields: `id` (string), `name` (string), `age` (number), `course` (string), `email` (string).
- **Student_List**: The component that displays all students in a tabular format.
- **Student_Form**: The Reactive Form component used for both adding and editing a student.
- **Search_Bar**: The UI control that accepts a keyword and filters the displayed student list.
- **Validator**: The Angular form-validation logic embedded in Student_Form.
- **Router**: The Angular Router that manages navigation between student views.
- **Toast_Service**: The notification service (from the Shared/UI module) used to display success and error messages.
- **Loading_Spinner**: The visual indicator (from the Shared/UI module) shown during pending HTTP requests.

## Requirements

### Requirement 1: Display Student List

**User Story:** As an administrator, I want to view a list of all students, so that I can see an overview of all enrolled students at once.

#### Acceptance Criteria

1. WHEN the administrator navigates to the `/students` route, THE Student_Module SHALL render the Student_List component.
2. WHEN the Student_List component initialises, THE Student_Service SHALL send a GET request to the Mock_API at the `/students` endpoint.
3. WHILE the GET request is pending and has been pending for no more than 10 seconds, THE Student_List SHALL display the Loading_Spinner.
4. WHEN the Mock_API returns a successful response with a non-empty Student array, THE Student_List SHALL render one table row per Student containing the student's `id`, `name`, `age`, `course`, and `email`.
5. WHEN the Mock_API returns a successful response with an empty Student array, THE Student_List SHALL render the table with zero rows and display a "No students found" message below the column headers.
6. IF the Mock_API returns an error response or returns data that cannot be parsed into a valid Student array, THEN THE Student_List SHALL hide the Loading_Spinner and display an error message stating that the student data could not be loaded.
7. WHEN the Student_List component renders, THE Student_List SHALL provide an "Add Student" button that navigates to the `/students/add` route.
8. WHEN the Student_List component renders a Student row, THE Student_List SHALL provide an "Edit" action for that row that navigates to the `/students/edit/:id` route for the selected Student.
9. WHEN the administrator clicks the "Delete" action for a Student row, THE Student_List SHALL display a confirmation prompt before sending any DELETE request.

### Requirement 2: Add Student

**User Story:** As an administrator, I want to add a new student record, so that newly enrolled students appear in the system.

#### Acceptance Criteria

1. WHEN the administrator navigates to `/students/add`, THE Student_Module SHALL render the Student_Form component in "add" mode.
2. THE Student_Form SHALL contain input fields for `name` (text, 2–100 characters), `age` (integer, 5–100 inclusive), `course` (text, 2–100 characters), and `email` (valid RFC 5322 format); the `id` field SHALL be auto-generated by the Mock_API and SHALL NOT be editable by the administrator.
3. WHEN the administrator submits the Student_Form and all four fields (`name`, `age`, `course`, `email`) satisfy their respective validation rules, THE Student_Service SHALL send a POST request to `/students` with the new Student payload containing `name`, `age`, `course`, and `email`.
4. WHILE the POST request is pending, THE Student_Form SHALL display the Loading_Spinner and disable the submit button.
5. WHEN the Mock_API returns a successful creation response, THE Router SHALL navigate to `/students` and THE Toast_Service SHALL display a success notification.
6. IF the Mock_API returns an error response, THEN THE Student_Form SHALL display an error notification via the Toast_Service without navigating away, and all previously entered field values SHALL be retained in the form.

---

### Requirement 3: Edit Student

**User Story:** As an administrator, I want to edit an existing student's information, so that I can correct or update student records.

#### Acceptance Criteria

1. WHEN the administrator navigates to `/students/edit/:id`, THE Student_Module SHALL render the Student_Form component in "edit" mode.
2. WHEN the Student_Form initialises in "edit" mode, THE Student_Service SHALL send a GET request to `/students/:id`.
3. WHEN the GET request returns a successful response, THE Student_Form SHALL pre-populate the `name`, `age`, `course`, and `email` fields with the values from the returned Student data and SHALL hide the Loading_Spinner.
4. WHILE the initial GET request is pending, THE Student_Form SHALL display the Loading_Spinner.
5. IF the GET request for `/students/:id` returns an error response, THEN THE Student_Form SHALL display an error notification via the Toast_Service.
6. IF the GET request for `/students/:id` returns an error response, THEN THE Router SHALL navigate to `/students`.
7. WHEN the administrator submits the Student_Form with all fields valid, THE Student_Service SHALL send a PUT request to `/students/:id` with the updated Student payload.
8. WHILE the PUT request is pending, THE Student_Form SHALL display the Loading_Spinner and disable the submit button.
9. WHEN the Mock_API returns a successful update response, THE Router SHALL navigate to `/students` and THE Toast_Service SHALL display a success notification.
10. IF the Mock_API returns an error response to the PUT request, THEN THE Student_Form SHALL hide the Loading_Spinner, re-enable the submit button, and display an error notification via the Toast_Service without navigating away.

---

### Requirement 4: Delete Student

**User Story:** As an administrator, I want to delete a student record, so that students who have left the system are removed.

#### Acceptance Criteria

1. WHEN the administrator clicks the "Delete" action for a Student, THE Student_List SHALL display a confirmation prompt that identifies the student by name and asks the administrator to confirm or cancel the deletion.
2. WHEN the administrator confirms the deletion, THE Student_Service SHALL send a DELETE request to `/students/:id` for the selected Student.
3. WHILE the DELETE request is pending, THE Student_List SHALL display the Loading_Spinner and the confirmation prompt's confirm button SHALL be disabled to prevent duplicate submissions.
4. WHEN the Mock_API returns a successful deletion response, THE Student_List SHALL remove the deleted Student row from the table and THE Toast_Service SHALL display a success notification.
5. IF the Mock_API returns an error response, THEN THE Student_List SHALL display an error notification via the Toast_Service and retain the Student row in the table.
6. WHEN the administrator cancels the confirmation prompt, THE Student_List SHALL take no action and close the confirmation prompt.

---

### Requirement 5: Search Students

**User Story:** As an administrator, I want to search students by keyword, so that I can quickly locate a specific student without scrolling through the entire list.

#### Acceptance Criteria

1. THE Student_List SHALL render a Search_Bar above the student table.
2. WHEN the administrator types a keyword of between 1 and 100 non-whitespace-only characters into the Search_Bar, THE Student_List SHALL filter the displayed rows to only those where the Student's `name`, `course`, or `email` contains the trimmed keyword (case-insensitive).
3. WHILE the Search_Bar contains a keyword, THE Student_List SHALL apply the filter to the in-memory student data without sending additional HTTP requests.
4. WHEN the administrator has not typed a new character for 300 milliseconds, THE Student_List SHALL apply the keyword filter to the displayed rows.
5. WHEN the Search_Bar is cleared or contains only whitespace, THE Student_List SHALL display all students within 100 milliseconds without any filter applied, bypassing the 300-millisecond debounce.
6. IF no students match the keyword, THEN THE Student_List SHALL display a "No students found" message in place of the table rows.
7. IF the administrator types a keyword exceeding 100 characters, THEN THE Search_Bar SHALL display a validation error message and THE Student_List SHALL not apply any filter.

---

### Requirement 6: Form Validation

**User Story:** As an administrator, I want the student form to validate my input before submission, so that invalid data is never saved to the system.

#### Acceptance Criteria

1. THE Validator SHALL require the `name` field to be non-empty and contain between 2 and 100 characters.
2. THE Validator SHALL require the `age` field to be a whole number between 5 and 100 inclusive.
3. THE Validator SHALL require the `course` field to be non-empty and contain between 2 and 100 characters.
4. THE Validator SHALL require the `email` field to match a valid email address format (RFC 5322 compatible).
5. WHEN focus leaves a field that fails its validation rule, THE Student_Form SHALL display an inline error message for that field describing the specific constraint that was violated.
6. WHEN the administrator activates the submit button while one or more fields have not yet been interacted with and are invalid, THE Student_Form SHALL display inline error messages for all invalid fields simultaneously.
7. WHEN one or more fields fail validation, THE Student_Form SHALL simultaneously display an inline error message for each invalid field describing the specific constraint that was violated.
8. WHILE any field in the Student_Form is invalid, THE Student_Form SHALL keep the submit button disabled.
9. WHEN the administrator corrects a field that previously failed validation, THE Student_Form SHALL clear the inline error message for that field.

---

### Requirement 7: Routing and Navigation

**User Story:** As an administrator, I want consistent navigation between student views, so that I can move between the list, add, and edit screens without losing context.

#### Acceptance Criteria

1. THE Router SHALL register the following routes for the Student_Module: `/students` (list), `/students/add` (add form), `/students/edit/:id` (edit form).
2. IF one of the three required routes fails to register, THEN THE remaining successfully registered routes SHALL still be accessible without error.
3. WHEN the administrator navigates to a URL that begins with `/students/` but does not match `/students`, `/students/add`, or `/students/edit/:id`, THE Router SHALL redirect to `/students`.
4. THE Student_Form SHALL provide a "Cancel" button that navigates to `/students` without submitting any data.
5. WHEN the administrator clicks "Cancel" on a Student_Form that contains unsaved edits, THE Router SHALL navigate to `/students` and all unsaved edits SHALL be discarded without any save or PUT/POST request being sent.
6. WHERE lazy loading is configured, THE Student_Module SHALL be loaded only when the administrator first navigates to a `/students` route in a given browser session.

---

### Requirement 8: API Integration

**User Story:** As a developer, I want all student HTTP operations to go through a single service, so that the API integration is centralised and easy to swap for a real back-end.

#### Acceptance Criteria

1. THE Student_Service SHALL expose the following public methods with the specified signatures: `getStudents(): Observable<Student[]>`, `getStudentById(id: string): Observable<Student>`, `createStudent(student: Omit<Student, 'id'>): Observable<Student>`, `updateStudent(id: string, student: Student): Observable<Student>`, `deleteStudent(id: string): Observable<void>`.
2. WHEN called, each Student_Service method SHALL return an RxJS `Observable` typed to the response shape defined in criterion 1 without any `any` type assertions.
3. THE Student_Service SHALL use Angular's `HttpClient` to communicate with the Mock_API base URL defined in the Angular environment configuration.
4. IF the Mock_API returns any HTTP error status code (4xx or 5xx) or if a network failure occurs, THEN THE Student_Service SHALL propagate the error through the Observable's error channel so that the calling component can handle it via an error callback or `catchError` operator.
5. THE Student_Service SHALL be provided in the application root injector so that it is available to all modules without re-instantiation.
